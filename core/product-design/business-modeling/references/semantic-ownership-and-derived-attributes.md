# 语义所有权、生命周期与派生属性判断

本参考用于修复“对象身份看似合理，但语义归属错误”的常见建模问题。

## 1. 对象身份异味

以下命名通常说明对象身份尚未收敛：

```text
Asset / Facility
User / Member
Request / Order
Team / Role / Vendor
```

斜杠、并列多个对象名或用一个对象同时承载不同生命周期，不应直接进入 confirmed 模型。处理方式：

1. 标记为 `provisional`；
2. 说明每个候选身份的边界和生命周期；
3. 评估是否需要拆分、上位对象或明确子类型；
4. 影响关系、权限、流程或数据库时进入 D2 人类决策。

## 2. 业务属性归属

### 2.1 固有业务属性

对象在不依赖某个具体对手方、时点或计算上下文时仍成立的语义，例如：设施的重要等级、人员的业务角色。

### 2.2 关系属性

只有两个对象发生特定关系时才成立，例如：某资源到某位置的到达成本、某成员在某工作区的权限。

关系属性应放在关系、候选对象、决策快照或专门的关联对象中，不应伪装成任一端对象的固有属性。

### 2.3 上下文 / 派生属性

由时点、来源、规则或其他事实计算得到，例如：当前剩余容量、SLA 风险、从当前位置到服务地点的预计到达时间。

必须说明：

- 派生来源；
- 计算或判断上下文；
- 是否需要持久化；
- 它是业务事实、决策证据还是下游实现建议。

## 3. 生命周期所有权

一个状态只能属于其对象自身的生命周期，并且对象必须已经存在。

示例：

```text
Dispatch Decision:
recommended → confirmed / overridden / rejected / cancelled

Assignment:
created → notified → accepted → active → finished
```

`recommended` 是派工决策的状态，不是 Assignment 的状态，因为责任承诺尚未成立。把另一个对象的前置判断状态放入当前对象，会导致对象身份、动作和审计责任混乱。

## 4. 一致性检查要求

`model-consistency-validator` 必须检查：

- 是否存在斜杠或多身份对象名；
- 关系 / 上下文属性是否错误挂在实体对象上；
- 对象首个状态是否早于对象创建动作；
- 状态转换是否引用稳定 Action ID；
- 决策、承诺、执行和证据对象的生命周期是否互相泄漏；
- 发现 D2 身份或边界问题时是否进入 Human Decision Control Plane。

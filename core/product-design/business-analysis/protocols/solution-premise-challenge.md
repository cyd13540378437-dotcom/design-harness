# Solution Premise Challenge / 解法前提挑战

## 1. 触发条件

当用户以具体解法提出问题，例如：

```text
我要做一个模拟器
我们应该降价吗
我要做企业版
是否要增加二级审批
```

不得默认该解法就是正确的问题表达。

## 2. 必须分离

```text
需要改善的业务结果
当前业务问题
用户提出的解法
维持现状
更小、更可逆的替代方案
延后或先验证
```

## 3. 分析顺序

```text
确认业务目标
→ 比较验证或解决路径
→ 形成是否采用该解法的商业判断
→ 用户确认后，才允许把实施细节作为附录或后续工作
```

## 4. 商业分析边界

商业分析可以说明某种产品或实验形态为什么值得做、能证明什么、不能证明什么、投入和风险是什么。

它不应在没有明确层级切换的情况下直接冻结：

```text
UI
数据 Schema
接口
技术架构
详细功能范围
算法实现
```
